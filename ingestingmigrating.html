Ingesting/Migrating

During ingests/migrations there are several items that has been covered by different sections of this document. It helps to have a working knowledge of linked data, RDF, and ontology & vocabulary basics vs repository-specific ontologies to create a comprehensive migration/ingest. Names, files, metadata along with the DOI is important to know how this information should be stored to process it correctly.

Batch

Ingesting Files according to the documentation the Migrate API is the main way to ingest batches of data into Drupal (and because Islandora 8 is Drupal, into Islandora). The Migrate module only provides the framework, it’s up to you to create the rules that take data from a source, through a process (i.e. a mapping) to a destination. A set of these rules is called a “migration”. It has to be set up (as a Configuration Entity, either by importing a YAML file or by installing a Feature) and then it has to be ran. In a nutshell the steps are the following

  1. Create a CSV file.
  2. Create a yml for each type of entity (person, place, event, university, etc) needing to mint.
  3. Create a “file” yml configuration file for the files.
  4. Create a “media” yml configuration file for the files to specify their media behavior once in Drupal.
  5. Create a “node” yml configuration file for gluing the file and media to a node. It the Islandora 7 terminology this would be the “object”.

Althought this process can be done in the admin UI, it’s highly suggested to use drush commands. The UI doesn’t respect order of operations and very little information is given from the admin UI when there is an error.
